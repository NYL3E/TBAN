Flappy TBAN — Déployer et exploiter une application web

Un jeu Flappy Bird avec un leaderboard en ligne, du code au monitoring en production.
Groupe : TBAN  ·  Membres : Lenny TRIDAT · Alexis BRUN · Raphaël ASSY · Seno NGUYEN
Application : Flappy TBAN (jeu + leaderboard)
Date : 18 juin 2026
Dépôt GitHub (public) : github.com/NYL3E/TBAN
URL publique (HTTPS) : https://tban.planbadge.fr

Seuls les étudiants mentionnés ci-dessus et visibles dans les commits du dépôt sont considérés comme contributeurs au projet.

Synthèse Flappy TBAN est un mini-jeu Flappy Bird jouable dans le navigateur, doublé d'un leaderboard persistant : chaque joueur enregistre son score, stocké dans une base PostgreSQL et affiché dans un classement « Top 10 ». L'objectif n'était pas de produire un jeu complexe, mais de démontrer la maîtrise du cycle de vie complet d'une application : développement → versionnement Git → conteneurisation Docker → déploiement Coolify → exploitation (monitoring, logs, mises à jour, sauvegardes, support). L'application est multi-conteneurs (un service applicatif Node.js + un service PostgreSQL avec volume persistant) et expose une route de santé /health utilisée à la fois par le Healthcheck Coolify et par Uptime Kuma.

1. Présentation du projet

Objectif de l'application Proposer un jeu d'arcade simple et addictif (Flappy Bird) avec une dimension compétitive : un classement en ligne des meilleurs scores, partagé entre tous les joueurs. Le projet sert de support pour illustrer un déploiement et une exploitation réels.
Public visé Tout utilisateur grand public (joueurs occasionnels) sur ordinateur ou mobile — l'interface est responsive et se joue à la souris, au clavier (Espace) ou au tactile.
Fonctionnalités principales
  • Écran d'accueil (menu) : pseudo, bouton « Jouer », classement et règles avant de lancer la partie.
  • Jeu Flappy Bird complet : gravité, tuyaux, collisions, score en temps réel.
  • Difficulté progressive : la vitesse augmente et l'ouverture se resserre doucement avec le score (effet borné, non punitif).
  • Décors dynamiques : l'ambiance change par paliers de score (jour, coucher de soleil, nuit étoilée…) avec un effet « niveau ».
  • Pseudo défini une seule fois (mémorisé via localStorage) → identité de joueur stable.
  • Sauvegarde automatique du score en fin de partie (API REST → PostgreSQL), sans action manuelle.
  • Leaderboard « Top 10 » en direct, avec une action de réinitialisation du classement.
  • Route de santé /health pour la supervision.
Donnée métier persistée Les scores des joueurs (table scores : pseudo, score, date). C'est cette donnée qui doit survivre aux redémarrages et être sauvegardée.
Solution d'IA intégrée ? Non — le produit n'embarque aucune fonctionnalité d'IA. (L'IA a uniquement servi d'assistant de développement, voir le Retour d'expérience.)

4. Déploiement

Comment l'application a été déployée

  1. Le code est poussé sur un dépôt GitHub public.
  2. Dans Coolify : New Resource → Docker Compose, dépôt connecté, branche main.
  3. Coolify lit le docker-compose.yml et construit les deux services app + db.
  4. Les variables d'environnement (POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB, DATABASE_URL) sont renseignées dans Coolify.
  5. Le domaine public est associé au service app (port 3000), HTTPS activé (Let's Encrypt).
  6. Le Healthcheck Coolify pointe sur /health → l'app passe en running:healthy.
Preuve technique — vérifications réelles sur l'application en production 
# Réponse HTTPS avec certificat Let's Encrypt valide
$ curl -sI https://tban.planbadge.fr/        ->  HTTP/2 200   (cert TLS valide)
# Route de santé : 200 + base de données joignable
$ curl -s  https://tban.planbadge.fr/health  ->  {"status":"ok","db":"up","uptime":504}
# Leaderboard : la donnée métier est bien persistée en PostgreSQL
$ curl -s  https://tban.planbadge.fr/api/scores
   ->  [{"name":"senio","score":96}, {"name":"ALEX","score":35}, ...]
Vérifié en direct sur l'environnement déployé : HTTPS valide, base connectée, persistance des scores confirmée via l'API. La validation d'entrée (HTTP 400 sur score invalide) a aussi été testée.

Technologies utilisées

Node.js 20, Express, pg (client PostgreSQL), PostgreSQL 16, Docker / Docker Compose, Coolify (Traefik + Let's Encrypt), Uptime Kuma.

Principales difficultés rencontrées

DifficultéSolution retenue
La base met quelques secondes à démarrer ; l'app plantait au boot. L'app démarre quand même (elle écoute tout de suite) et retente la connexion à la base (15 tentatives). /health renvoie 503 tant que la base n'est pas prête, puis 200.
Ordre de démarrage des conteneurs. depends_on: condition: service_healthy + pg_isready sur le service db.
Persistance de la donnée entre deux déploiements. Volume nommé db_data monté sur /var/lib/postgresql/data (à ne jamais supprimer).
Erreur Traefik « 503 no available server » au 1ᵉʳ déploiement : le domaine était routé vers le port 80 par défaut, alors que l'app écoute sur 3000. Dans Coolify, le domaine doit cibler le port interne du conteneur : on l'a saisi sous la forme https://tban.planbadge.fr:3000. Résolution immédiate → app accessible + HTTPS Let's Encrypt émis.

5. Exploitation

5.1 Supervision (monitoring + alerting)

Idée clé. Monitoring = savoir si le service fonctionne. Alerting = prévenir quelqu'un quand il ne fonctionne plus. Nous combinons deux niveaux :

DispositifConfigurationCe qu'il vérifie
Healthcheck CoolifyPath /health, port 3000, statut attendu 200. Interne : l'app et la base répondent (la route teste SELECT 1).
Uptime KumaMonitor HTTP(s) sur l'URL publique, intervalle 60 s, nom tban-flappy. Externe : le service est réellement accessible « comme un utilisateur ».
Healthcheck DockerDéfini dans le Dockerfile et le compose. Le conteneur se déclare lui-même sain / malsain auprès de Docker.

En cas d'incident : Uptime Kuma passe au rouge et peut notifier (e-mail, Discord, Telegram…). En entreprise, l'alerte irait à la personne d'astreinte (ops / dev responsable du service).

📷 Capture — Healthcheck Coolify healthy + monitor Uptime Kuma au vert (Up).

5.2 Logs

Où : Coolify → application → onglet Logs (logs de build + logs du conteneur, sans SSH). Lignes types :

[server] Flappy TBAN à l'écoute sur le port 3000
[db] Connecté à PostgreSQL — table "scores" prête.
[api] Nouveau score enregistré : Lenny = 27
[health] Base de données injoignable : ...   (symptôme : la base est tombée)

Utilité : la 2ᵉ ligne confirme un démarrage réussi (app + base) ; la 3ᵉ trace l'activité métier (un score a été enregistré) ; la dernière apparaît si la base devient injoignable — c'est le 1ᵉʳ endroit où regarder pour diagnostiquer. Pour retrouver le problème d'un utilisateur, on lui demande l'heure exacte, l'URL, l'action et le navigateur afin de filtrer la bonne fenêtre de logs.

5.3 Stratégie de mise à jour

  • Briques à maintenir : image Node 20, PostgreSQL 16, Coolify/Traefik, l'OS hôte, et nos dépendances (express, pg).
  • Process : modification → commit GitHub → Coolify redéploie automatiquement ; possibilité de rollback vers un déploiement précédent depuis Coolify.
  • Vérification post-MAJ : Healthcheck toujours healthy + test manuel d'une partie + envoi d'un score de test + surveillance des logs et d'Uptime Kuma.
  • Compromis : ne jamais mettre à jour = failles non corrigées ; tout mettre à jour sans tester = risque de régression. On sauvegarde avant toute mise à jour de la base.

5.4 Stratégie de sauvegarde (vue d'ensemble)

Le code est sauvegardé par Git (GitHub + clones locaux). La base de données est sauvegardée par pg_dump planifié. Détail et application de la règle 3-2-1 ci-dessous.

Commande de sauvegarde 
$ docker compose exec db pg_dump -U flappy flappy > backup_$(date +%F).sql
# Restauration :
$ cat backup_2026-06-18.sql | docker compose exec -T db psql -U flappy flappy
Phrase à retenir : « Une sauvegarde non testée est une hypothèse. » Une restauration doit être testée au moins une fois pour être considérée comme fiable.

7. Support utilisateur

Face à un utilisateur qui signale « votre application ne fonctionne pas », l'objectif n'est pas de répondre « ça marche chez moi » mais de collecter les bonnes informations pour retrouver la trace dans les logs.

Réponse type au ticket

Objet : Re : Problème d'accès à Flappy TBAN — quelques informations pour vous aider

Bonjour,
Merci pour votre signalement, nous allons vous aider rapidement. Pour identifier la cause, pourriez-vous nous préciser :

  • l'URL exacte utilisée ;
  • l'heure approximative du problème (et votre fuseau) ;
  • une capture d'écran de l'erreur ;
  • le navigateur et l'appareil (ex. Chrome sur Windows 11, Safari sur iPhone) ;
  • les étapes réalisées juste avant l'erreur (ex. « au moment d'enregistrer mon score ») ;
  • le message d'erreur exact, s'il y en a un.

Ces éléments nous permettront de retrouver la trace correspondante dans nos logs.
Merci d'avance,
L'équipe TBAN

Prise en main à distance (si nécessaire et avec l'accord de l'utilisateur) : Assistance rapide / Partage d'écran macOS, RustDesk, AnyDesk ou TeamViewer.

8. Retour d'expérience

Ce qui a bien fonctionné Le workflow Git en équipe : chacun a développé sa fonctionnalité sur sa branche (page d'accueil, décors dynamiques) puis l'a intégrée par Pull Request et merge. L'enchaînement Git → Docker → Coolify redéploie tout en un push, et la route /health a rendu le monitoring concret et fiable.
Ce qui a été difficile Le routage Coolify (indiquer le port du conteneur dans le domaine), le timing app/DB au démarrage, la persistance par volume, et la résolution des conflits Git en fusionnant les branches de chacun.
Ce que nous améliorerions Ajouter une route /metrics et un vrai dashboard métier (cf. indicateurs ci-dessous), anti-triche côté serveur sur les scores, sauvegardes automatisées et planifiées avec restauration testée, et alerting Uptime Kuma branché sur un canal d'équipe.
Outil d'IA utilisé (assistance au développement) Le produit n'embarque pas d'IA. En revanche, nous avons utilisé Claude (Claude Code, modèle Anthropic Opus) comme assistant de développement (génération de code, Dockerfile, documentation), tout le code ayant été relu et testé par le groupe.

9. Indicateurs métier (bonus exploitation)

Une application peut être techniquement en ligne (serveur OK, HTTPS OK) sans remplir son objectif. Trois KPIs utiles pour Flappy TBAN, tous calculables depuis la table scores :

  1. Nombre de parties enregistrées / jour — volume d'usage réel (COUNT(*) par jour).
  2. Score moyen et meilleur score — engagement et niveau des joueurs (AVG, MAX).
  3. Nombre de joueurs distincts — audience (COUNT(DISTINCT name)).
